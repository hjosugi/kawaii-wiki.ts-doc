# HTTP API契約

kawaii-wiki.ts 1.xでは、`/api`以下のHTTPルートを公開インテグレーションのインターフェースとして扱います。
インタラクティブなドキュメントは`/api/docs`で提供され、生成されたOpenAPIドキュメントは`/api/docs/openapi.json`で入手可能です。
Elysiaルートのランタイムスキーマと`@kawaii-wiki/server`でエクスポートされるTypeScript契約が真の情報源となります。

GraphQLはまだ1.x契約の一部ではありません。インテグレーションはRESTルートを使用すべきであり、これによりロールチェック、ページルール、レート制限、監査動作がウェブアプリと同一に保たれます。

## 安定性ポリシー

1.xリリースライン内では：

- 既存のルートメソッド、必須リクエストフィールド、およびレスポンスフィールドは削除または名前変更されません；
- 新しいオプションのリクエストフィールド、レスポンスフィールド、ルート、および列挙型の値はマイナーリリースで追加されることがあります；
- セキュリティ修正により、常に無効または安全でなかった入力が拒否される場合があります；
- 廃止予定は、次のメジャーリリースでの削除前に少なくとも1つのマイナーリリースの間、変更履歴に記載されます。

クライアントはまだ理解していない未知のレスポンスフィールドや列挙型の値を無視すべきです。ブラウザ専用セットアップ、OAuthコールバック、リアルタイム、および静的アセットのルートは、テストはされているもののOpenAPIドキュメントから除外される場合があります。

APIキーは`Authorization: Bearer <key>`を使用し、割り当てられたロールによって制限されます。セッションJWTも同じヘッダーを使用しますが、バンドルされたウェブアプリ向けです。いずれの認証情報もURLに含めてはいけません。Webhookペイロードには独自の明示的な`version`フィールドとHMAC署名契約があります。

破壊的変更は新しいメジャーアプリケーションリリースを必要とします。バージョン管理されたWebhookペイロードは、新しいペイロードバージョンを独立して導入でき、古いサポートされているバージョンは変更履歴に記載されたままです。
